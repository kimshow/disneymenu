## 📄 主要ドキュメントの説明

### ルートレベルドキュメント

#### README.md
**目的**: プロジェクトの概要と基本情報  
**対象読者**: すべてのユーザー（開発者、エンドユーザー、コントリビューター）  
**更新頻度**: 機能追加時、技術スタック変更時  
**主な内容**:
- プロジェクト概要と主要機能
- 技術スタック
- セットアップ手順
- API エンドポイント一覧
- 開発ロードマップ

#### DEVELOPMENT.md
**目的**: 開発環境のセットアップと開発ガイドライン  
**対象読者**: 開発者  
**更新頻度**: 開発ワークフロー変更時  
**主な内容**:
- ローカル開発環境構築
- テスト実行方法
- コーディング規約
- Git ワークフロー

#### DEPLOYMENT.md
**目的**: デプロイ手順と本番環境管理  
**対象読者**: DevOps、プロジェクトマネージャー  
**更新頻度**: デプロイプロセス変更時  
**主な内容**:
- Vercel デプロイ設定
- 環境変数管理
- CI/CD パイプライン

#### SECURITY.md
**目的**: セキュリティポリシーと脆弱性報告ガイドライン  
**対象読者**: セキュリティ研究者、コントリビューター  
**更新頻度**: 半年に1回、または重大な変更時  

#### STARTUP.md
**目的**: 最速でプロジェクトを起動するためのクイックガイド  
**対象読者**: 新規開発者  
**更新頻度**: セットアップ手順変更時

## 🔄 ドキュメント更新フロー

### 機能追加時

1. **設計書作成**: `docs/PHASE{N}_PLAN.md` を作成
2. **実装中**: コード内コメントを充実させる
3. **完了後**: README.md の機能一覧を更新
4. **テスト完了後**: TESTS.md にテストケース追加
5. **Phase完了後**: 設計書を `docs/archives/` に移動

### API変更時

1. `docs/API_REFERENCE.md` を更新
2. OpenAPI スキーマ（FastAPI）を更新
3. フロントエンドの型定義を更新
4. README.md のAPIエンドポイント一覧を更新

### 技術スタック変更時

1. README.md の技術スタック一覧を更新
2. DEVELOPMENT.md のセットアップ手順を更新
3. package.json / requirements.txt を更新
4. DEPLOYMENT.md のデプロイ手順を確認

## 📝 ドキュメント作成ガイドライン

### マークダウン規約

#### ヘッダー
```markdown
# レベル1: ドキュメントタイトル（1つのみ）
## レベル2: 大セクション
### レベル3: 中セクション
#### レベル4: 小セクション
```

#### コードブロック
````markdown
```typescript
// 言語名を明記
const example = "code";
```
````

#### リンク
```markdown
[表示テキスト](相対パス or URL)

# 相対パスの例
[API Reference](docs/API_REFERENCE.md)

# URLの例
[Material-UI](https://mui.com/)
```

#### 表
```markdown
| ヘッダー1 | ヘッダー2 |
|----------|----------|
| データ1  | データ2  |
```

### 絵文字の使用

**推奨される使用箇所**:
- セクションヘッダー（📋 目次、🎯 目標、⚠️ 注意）
- ステータス表示（✅ 完了、🚧 進行中、❌ 未着手）
- 優先度（🔥 高、🟡 中、🔵 低）

**使用禁止箇所**:
- コードブロック内
- ファイルパス
- 技術用語
